§ Empty / null field rules (static mock shows every value — real data won't)

  • Inline value missing (phone, ref, customer_so_no): render an em dash — in --mut2, keep the row.
  • Whole optional row missing (emergency contact, note): hide the row entirely — don't show a blank.
  • Optional section missing (deposit/押金 段, disposal, setup photos): collapse the whole card — see per-screen "only when" notes.
  • Amount missing / not yet computed: — not RM 0.00 (0.00 means genuinely zero, e.g. fully paid → "No balance — fully paid").
  • Address blocks: if address1 empty but SO flagged "fill later" → show the amber "Fill in address later" note instead of blank lines.

§ Variable-count blocks (item 21)

  • 0 rows: show the screen's empty state (§ per screen).
  • 1 row: single card, no "N items" counter needed (or "1 line").
  • Many: vertical stack, gap 11px; the card header shows a count ("N lines").
  • Overflow — chips / filters / folder tabs / stage pipeline: horizontal scroll (overflow-x:auto, hidden scrollbar), never wrap.
  • Overflow — line items / payments: vertical, no cap; long lists paginate/lazy-load server-side.

Document lifecycle — View · Edit · Cancel · Convert · Draft/Save

One universal pattern for all transactional docs (SO · Consignment · Returns · PO · GRN · Delivery · Service · Project). The matrix below gives each module's specifics.

The five states (same screen, different mode)

  • 1 · VIEW — opening a saved doc shows the create form, locked: every card + field from New, rendered read-only (.fld-ro), with a KPI strip on top (Total/Paid/Balance where money applies). Never a separate summary. Action bar = status-dependent (below).
  • 2 · EDIT — tapping Edit swaps .fld-ro → .fld-i in place (no new screen); server-generated fields (doc_no, timestamps) stay locked. Action bar becomes [Discard] [Save changes]. Only allowed in editable statuses (see matrix) + with the module's edit permission.
  • 3 · CANCEL / VOID — destructive action → confirm dialog (§ overlays): "Cancel this order? …". On confirm: PATCH …/cancel, status → CANCELLED/VOID, action bar hides, doc renders greyed (55% opacity) + a "Cancelled {{cancelled_at}}" ribbon. Permission-gated.
  • 4 · CONVERT — forward the doc to the next document type via the Convert wizard (bottom-sheet, 3 steps: pick lines+qty → assign → confirm). Each module's target(s) are in the matrix (SO→DO, DO→SI, PO→GRN, SO→PO, Consignment→Invoice…). Only outstanding qty is convertible; success → new doc + toast + deep-link.
  • 5 · DRAFT / SAVE — a new/unsubmitted doc is DRAFT: action bar [Save draft] [Submit]. Save draft = POST/PATCH keeps status DRAFT (stays editable, no doc_no shown as "Assigned on save"). Submit = validate → status SUBMITTED (or CONFIRMED per module) → becomes the locked VIEW.

  • Input: name, phone, email, customer_so_no, building_type, dates, note, line variants, qty, payments.
